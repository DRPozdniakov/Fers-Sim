High-Level Design
System Overview
KDCube processes chat messages through a queue-based pipeline. Clients connect via SSE or Socket.IO, messages are enqueued into Redis, workers execute dynamic bundle workflows, and events stream back through a Redis Pub/Sub relay.

Knowledge Base
1

Enqueue

Gateway validates auth, enforces rate limits, and pushes the chat task into a priority Redis queue (privileged / registered / anonymous).

2

Dequeue + Lock

Processor workers BRPOP fairly across queues, acquire a per-task concurrency lock, and begin execution.

3

Execute Bundle

The selected dynamic bundle runs: ReAct solver, tool calls, code execution, memory retrieval, and LLM inference.

4

Publish Events

Bundle emits chat_start, chat_delta, chat_step, chat_complete events to the Redis Pub/Sub relay channel.

5

Fan-out

The relay forwards events to all API instances subscribed to the session channel.

6

Stream to Client

Gateway pushes events to the connected client via SSE or Socket.IO with role-based filtering.

Deep Dive
Authentication Flow
Production uses cookie-only auth with infra exchange. The client never sees real tokens — Nginx handles the ProxyLogin exchange and sets secure HTTP-only cookies.
🔒
Security model: The client only ever holds a masked (non-real) cookie. Real access/id tokens are exchanged server-side by Nginx via ProxyLogin. The API accepts tokens from headers, cookies, SSE query params, or Socket.IO auth payload for backwards compatibility.
Deep Dive
Streaming & Relay Architecture
Both SSE and Socket.IO share the same Redis Pub/Sub relay, scoped per session. The design achieves low fan-out cost with no global firehose, and full transport symmetry.
⚡
Transport symmetry: SSE and Socket.IO are fully interchangeable. The relay doesn't know which transport the client uses — it just publishes to the session channel, and the appropriate hub delivers events.
Deep Dive
Bundles & Runtime
Bundles are runtime-loadable agentic workflows. Each request selects a bundle, and the bundle orchestrates ReAct solving, tool execution, code execution, and memory management. Multiple bundles can coexist; one executes per request.
1

Bundle Selection

Each message specifies which bundle to use. The registry loads and caches the bundle's entrypoint, endpoints, and config.

2

ReAct Planning

The solver plans the approach: tool-first, code-first, or direct inference. It adaptively selects skills and tools.

3

Execution Loop

Tools run locally or via MCP, code executes in ephemeral Docker containers, and memory is retrieved from Postgres/S3.

4

Event Streaming

ChatCommunicator emits events through the channeled streamer with step boundaries, deltas, citations, and cost envelopes.

Deep Dive
Economics & Rate Limiting
Two-layer protection: gateway-level rate limits (requests/hour) and economics-level tier-aware quotas (budgets, concurrency locks, per-user limits).
1

Gateway Check

Request rate limits, backpressure (queue capacity), and circuit breakers are evaluated. Rejected requests get 429.

2

Economics Check

Tier-aware quotas, budget limits, and concurrency locks are enforced. Over-budget users get 402 or 429.

3

Execute

Request proceeds to the workflow bundle for processing if both checks pass.

4

Track & Update

Usage/cost is tracked per-request and reported to the control plane for budget deduction and spending reports.

Deep Dive
Horizontal Scaling
The architecture supports horizontal scaling through stateless API instances and independent processor workers, connected by Redis for coordination and relay.
⚙
Scaling model: API instances are stateless and scale behind a load balancer. Each maintains SSE/Socket.IO connections and subscribes to Redis relay channels. Processor workers are independent — they BRPOP fairly across priority queues and respect max-concurrent limits. Add more instances of either layer independently.
